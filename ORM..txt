ORM (Object-Relational Mapping) is a technique that lets you interact
 with a database using Python code instead of SQL queries.
Django ORM allows us to perform database operations using Python objects 
instead of writing raw SQL queries

Why we use ORM
No need to write SQL
Database independent (SQLite, MySQL, PostgreSQL)
Cleaner & faster development
Safer (prevents SQL injection)

Database table → Python class (Model)
Table row → Object
Columns → Fields


=>
A model is a Python representation of a database table.

👉 It defines:

What data you store
Structure of the table (columns/fields)

=>
A model class is the Python class you create by inheriting models.Model.

👉 In the example:

class Student(models.Model):
Student = Model class
It defines the structure of the database table

"A model class in Django defines the structure of a database table.
 It provides the schema, such as fields and their data types, and Django 
 uses it to create the corresponding table


 MakeMigrations VS Migrate VS ShowMigration

 In Django, **`makemigrations`**, **`migrate`**, and **`showmigrations`** are all part of the migration system—but they serve very different purposes. Think of them as *create → inspect → apply*.

---

## 1. `makemigrations` 🛠️

**What it does:**
Creates new migration files based on changes in your models.

**How it works:**
Django compares your current models with the last migration state and generates a new migration file.

**Example:**

```bash
python manage.py makemigrations
```

**Use when:**

* You add/remove fields
* Change field types
* Create/delete models

**Output:**
Creates files like:

```
app_name/migrations/0002_auto_xyz.py
```

👉 Important: It **does NOT touch the database**, only creates migration files.

---

## 2. `migrate` 🚀

**What it does:**
Applies migrations to the database.

**How it works:**
Reads migration files and executes SQL to update the database schema.

**Example:**

```bash
python manage.py migrate
```

**Use when:**

* After running `makemigrations`
* When setting up a new database

**What it affects:**

* Tables
* Columns
* Constraints

👉 This is the command that actually **modifies your database**.

---

## 3. `showmigrations` 👀

**What it does:**
Displays all migrations and whether they’ve been applied.

**Example:**

```bash
python manage.py showmigrations
```

**Output example:**

```
app_name
 [X] 0001_initial
 [X] 0002_auto_xyz
 [ ] 0003_new_field
```

* `[X]` → applied
* `[ ]` → not applied

👉 Useful for debugging migration issues.

---

## 🔁 How they work together

Typical workflow:

```bash
# 1. Make changes in models.py
python manage.py makemigrations

# 2. Apply those changes to DB
python manage.py migrate

# 3. Check status (optional)
python manage.py showmigrations
```

---

## 🧠 Simple analogy

* `makemigrations` → Write instructions (like a blueprint)
* `migrate` → Execute instructions (build the structure)
* `showmigrations` → Check what’s already built
